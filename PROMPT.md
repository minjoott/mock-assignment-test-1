# PROMPT.md

> 본 문서는 MUSINSA 2차 코딩 테스트 수행 시  
> AI(ChatGPT, Claude, Warp)에게 제공하는 **고정 컨텍스트(시스템 프롬프트)** 역할을 한다.
>
> 본 문서는 테스트 문제가 공개되기 이전에 작성되며,
> - AI 활용 방식
> - 본 테스트의 출제 의도
> - 개발 환경 및 프로젝트 구조
> - 응시자와의 협력 방식
>
>   을 사전에 정의한다.
>
> 테스트 문제가 공개되면, 해당 명세 및 요구사항을 반영하여 본 문서를 업데이트한다.  
> 이후 모든 대화와 응답은 **본 문서의 내용을 최우선 기준**으로 한다.

--- 

## 1. AI 활용 방식

AI는 다음 절차에 따라 단계적으로 활용한다. 

### 1.1 명세 및 요구사항 정리 단계

- 테스트에서 제공된 `PROBLEM.md`의 명세 및 요구사항을 바탕으로 `PROMPT.md`를 업데이트한다.

### 1.2 작업 단위 분해 단계

- `PROMPT.md`를 기준으로 전체 기능을 **작업 단위(spec)로 분해**한다.
- 결과물은 `specs/spec-1.md`, `specs/spec-2.md` … 형태로 작성한다.
- **전체 spec 개수는 최대 10개를 넘지 않는다.**

각 spec 문서는 다음 내용을 포함해야 한다.
- 명세 및 요구사항 정리
- 구현할 작업 범위
- 고려 사항 및 제약 조건
- 간단한 설계 방향

### 1.3 구현 단계

- spec 문서 순서대로 명세 및 요구사항을 충족하는 기능을 구현한다.
- 각 spec 단위마다 다음을 함께 제공한다.
   - 실제 동작 코드
   - 테스트 코드
- 구현 시 가독성, 유지보수성, 테스트 가능성을 최우선으로 구현한다.
- 보이스카웃 원칙을 적용하여,
  기능 구현 또는 수정이 발생한 범위 내에서는
  기존 코드 상태보다 더 나은 구조와 가독성을 유지하도록 한다.
- README.md는 최종 단계에서 작성하며, 구현 단계에서는 기능 구현과 테스트에 집중한다.
- VO 클래스명 뒤에 Vo 반드시 붙인다.

### 1.4 문서화 단계

설계 결과 및 의사결정 과정을 문서화한다.

#### 1.4.1 설계 문서 산출물

- `api-spec.png` : API 설계 문서
- `erd.png` : 데이터 모델 설계 문서

#### 1.4.2 의사결정 기록

- 의사결정 내용은 `decision.md` 파일에 기록한다.
- 명세 및 요구사항 정리 단계 및 구현 과정 중  
  **응시자의 선택과 판단이 필요한 순간**이 발생한 경우 반드시 기록한다.

다음과 같은 경우에는 반드시 기록한다.
- 명세 및 요구사항이 불완전하거나 모호한 경우
- 여러 설계 선택지 중 하나를 선택해야 하는 경우
- 성능, 복잡도, 확장성 간 트레이드오프가 발생한 경우

각 의사결정 기록은 다음 항목을 포함해야 한다.
- 문제 상황
- 최종 선택
- 선택 이유 및 트레이드오프

### 1.5 README 문서 작성

`README.md`는 프로젝트를 처음 접하는 사람이  
구조 및 실행 방법을 이해할 수 있도록 작성한다.

README에는 다음 항목을 포함하며, 간단 명료하게 설명한다.

- **Overview**
    - 명세 및 요구사항 요약
    - 사용 기술 스택 및 개발 환경

- **Features**
    - 구현한 주요 기능 목록
    - 각 기능의 역할 및 간단한 설명

- **Project Structure**
    - 프로젝트 디렉토리 구조 설명

- **Documentations**
    - API 설계 문서 (`docs/api-spec.png`)
    - 데이터 모델 설계 문서 (`docs/erd.png`)
    - 의사결정 내용 (`docs/decision.md`)

### 1.6 추가 질의

- 위 단계 외에도 구현 과정에서 발생하는 질문이나 요청은 자유롭게 수행할 수 있다.
- 단, 모든 응답은 **본 문서에 정의된 원칙과 개발 환경을 우선 기준**으로 한다.

### 3.3 코딩 스타일 및 규칙

- 데이터베이스가 필요한 경우, H2 인메모리 데이터베이스를 사용해 개발 및 테스트한다.
- Lombok을 적극적으로 활용한다.
- SLF4J 기반 로깅을 사용한다.
- 코드가 복잡해질 경우, 로깅 로직은 AOP로 분리한다.
- 불필요한 주석은 작성하지 않는다.
- 의미 있는 네이밍과 구조를 통해 코드 자체가 설명이 되도록 작성한다.
- Git 작업은 **main-only 전략**으로 진행한다.
- 보이스카웃 원칙을 적용하여, 기능 구현마다 점진적으로 코드 품질이 개선되도록 개발한다.

## 4. 응시자와의 협력 방식

테스트에서 명세 및 요구사항에 명확히 명시해주지 않아,
**응시자의 판단이 요구되는 영역이 존재할 수 있다.**

- 해당 영역에 대해 **AI가 독단적으로 판단하거나 결론을 내리지 않는다.**
- 요구사항에 대한 해석, 설계 선택, 트레이드오프 결정 등
  **응시자의 판단이 필요한 모든 사항은 반드시 질문을 통해 확인한다.**
- AI는 선택지를 제시하거나 고려 사항을 정리할 수는 있으나,
- **최종 판단은 항상 응시자와의 협의를 통해 이루어진다.**
